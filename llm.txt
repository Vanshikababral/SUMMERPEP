What are we doing?
Pipeline: Input → Prompt → Model → Parser → Output
We first provide an input to the application.
That input is converted into a prompt (either a Single Message Prompt or a Multi-Message Prompt).
The prompt is then sent to the LLM (Model).
The model generates a response.
The response is passed to an Output Parser, which formats or extracts the required information.
Finally, the parsed result is displayed as the output.

Here are the steps in a short and easy-to-remember format:

(i)Import the required libraries.
(ii)Create the Prompt Template (ChatPromptTemplate).
(iii)Create the Model (ChatGoogleGenerativeAI).
(iv)Create the Output Parser (StrOutputParser).
(v)Invoke the Prompt using chat_prompt.invoke() to generate the final prompt.
(vi)Invoke the Model using model.invoke() to get the model's response (AIMessage).
(vii)Invoke the Parser using parser.invoke() to extract the text from the model's response.
(viii)Print the final output.

****Conditional Chaining****
RunnableBranch is used to implement conditional chaining in LangChain. It allows you to execute different chains based on a condition.

from langchain_core.runnables import RunnableBranch

conditional_chain = RunnableBranch(
    (func1, chain1),
    (func2, chain2),
    default_chain
)
How it works
Each function (func1, func2, ...) acts as a condition and must return either True or False.
RunnableBranch evaluates the conditions from top to bottom.
As soon as a condition returns True, the corresponding chain is executed.
The remaining conditions are not checked.
If none of the conditions return True, the default_chain is executed (if provided).
Example Flow
            Input
              |
              |
     -------------------
     |                 |
   func1?           False
     |
   True
     |
   chain1
     |
    Output

If func1 = False
        |
      func2?
     /     \
  True     False
   |          |
chain2   default_chain
   |          |
 Output     Output
Key Points
RunnableBranch works like an if → elif → else statement in Python.
Conditions are evaluated sequentially.
The first matching condition determines which chain is executed.
A default_chain is optional but recommended to handle cases where no condition matches.

---Chunking----
df: Chunking is the process of breaking a large document into smaller pieces (chunks) before storing it in a vector database

Why chunking? 
retrieval becomes much easier

We try to split the documnents into several chunks .
Advantages:
    -Performance
    -To optimize cost and resources 
    -Context window limit 

----------Different ways to perform chunking:------------

1. Fixed Size Chunking- Break the document after a fixed number of characters
    Advantages
        Very simple
        Fast
    Disadvantages
        May split a sentence in the middle
2. Recursive Chunking

Paragraph
↓
Sentence
↓
Word
↓
Character

It first tries to split at paragraph boundaries.
If the paragraph is still too large,
it splits into sentences.
If the sentence is too long,
it splits into words.
Only as a last resort does it split by characters.

3. Sentence-Based Chunking - Split after every sentence.

4. Paragraph-Based Chunking - Split after every paragraph.

5. Token-Based Chunking - Instead of characters,split according to tokens.

6. Semantic Chunking - This is an advanced technique.Instead of splitting by size,it splits whenever the topic changes.
   Semantic chunking usually produces better retrieval quality but is more computationally expensive.

-------Retrievers--------

WData source -> Relevant documents 
Therefore we query to find the relevant information

https://medium.com/@abhishekjainindore24/langchain-part-12-retrievers-b435ae8f4ea7

Earlier, we stored documents in a vector database and retrieved them using similarity search. However, not all data needs to be stored in a vector database. A retriever can also fetch relevant documents directly from external sources like Wikipedia or Arxiv at query time. That's why LangChain provides different types of retrievers depending on where the data is stored.

Top three retrievers:-
  (i)MMR
  (ii)MQR
  (iii)CCR

-> MMR (Maximum Marginal Relevance)
Problem: Normal similarity search may return duplicate or very similar documents.
Example:
Query: Causes of Climate Change

Doc1 → Arctic glaciers are melting.
Doc2 → Ice caps are melting.
Doc3 → Greenhouse gases increase global warming.

A normal retriever might return:
Doc1 ✅
Doc2 ✅ (almost the same as Doc1)
Doc3

MMR returns:
Doc1 ✅
Doc3 ✅
Another different but relevant document

Goal: Return relevant + diverse documents.

-> MQR (Multi-Query Retriever)
Problem: A user may ask an ambiguous question.

Example:
Query:
How can I stay healthy?

The LLM rewrites it into multiple queries:

How to eat healthy?
How to exercise regularly?
How to improve mental health?
How to sleep better?

Each query searches the vector database, and all results are merged.

Goal: Improve retrieval by searching from multiple perspectives.

-> CCR (Contextual Compression Retriever)
Problem: Retrieved documents often contain unnecessary information.
Example:

Retrieved Document:
The Grand Canyon is famous.
Photosynthesis converts sunlight into energy.
Millions of tourists visit every year.

The user asked:
What is photosynthesis?

CCR compresses the document to:
Photosynthesis converts sunlight into energy.

Goal: Remove irrelevant text and keep only what's needed.
